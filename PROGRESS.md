- ✅ 버그 수정(부분): 한글 닉네임 입력 중 마지막 글자가 사라지는 문제 — `MainPage.tsx`의 닉네임 `<input>`이 `onChange`(native `input` 이벤트)에만 의존해, IME 조합이 `input` 이벤트 없이 `compositionend`로만 끝나는 경우 React state가 갱신되지 않고, 이후 다른 상태 변화로 리렌더될 때 컨트롤드 인풋이 오래된 state로 DOM을 덮어써 글자가 사라짐. `onCompositionEnd`와 `onBlur`에서 실제 DOM 값을 state에 명시적으로 재동기화하도록 수정 — "같은 페이지 안에서 다른 입력으로 포커스 이동" 케이스는 Playwright로 재현·수정 확인 완료. 단, "브라우저 탭 전환 단축키와 마지막 키 입력이 정확히 겹쳐 그 키 자체가 브라우저 크롬으로 소비되는" 레이스 컨디션은 웹페이지 JS로 근본 해결 불가능한 브라우저/OS IME 레벨 이슈로 확인됨(대형 한국 사이트에서도 동일 현상 재현되는 것을 사용자가 확인) — 이 부분은 프로젝트 코드로 추가 조치하지 않기로 함.
- ✅ `frontend/index.html`의 `<html lang="en">` → `lang="ko"`로 변경 — 크롬이 페이지를 영어로 오인해 새로고침 시 번역 팝업이 뜨던 문제 해결 (IME 버그와는 무관, 별개 개선).
- ✅ 비주얼 리디자인 (파스텔 캐주얼 + 카드 게임 연출): Tailwind CSS v4(`@tailwindcss/vite`) 도입, `index.css`에 라이트/다크 파스텔 컬러 토큰(`--primary`/`--secondary`/`--peach`/`--danger` 등)을 `@theme inline`으로 매핑. 향후 다른 게임 추가를 대비해 `components/ui/`(Button, Card, Badge, PlayerAvatar) 공유 레이어와 `game/rps/`(백엔드 `game/rps/`와 대칭) 전용 레이어로 분리. 가위바위보를 버튼이 아니라 **카드를 내는 형식**으로 재구성: `game/rps/cards.ts`에 `Hand` → 표시 정보(이모지/라벨) 매핑을 별도 데이터 테이블(`CardDef`)로 분리해, 나중에 특수카드를 추가할 때 이 파일만 확장하면 되도록 설계함(단, 이번 작업은 비주얼까지만 — 특수카드 판정 로직/백엔드는 손대지 않음, `types/index.ts`의 `Hand` 유니온도 그대로 유지). `game/rps/components/HandCard.tsx`(카드 한 장, face-up/face-down/revealed) + `PlayZone.tsx`(나·상대 대결 슬롯, 동시 flip 리빌)로 `GamePage.tsx`를 재구성, `RoomPage.tsx`/`MainPage.tsx`도 공유 UI로 재스타일링.
  - 🐛 구현 중 발견: CSS 3D 카드 플립에서 `.flip-card-back`/`.flip-card-front`의 정적 `rotateY(180deg)`를 반대쪽에 걸어서, 애니메이션이 끝나면 오히려 뒷면(미스터리 패턴)이 보이고 실제 카드 앞면이 숨는 버그가 있었음. Playwright로 결과 화면을 캡처했을 때 텍스트가 뜨자마자(트랜지션 시작 직후, 0ms 근처) 스크린샷을 찍으면 우연히 정상처럼 보여서 처음엔 놓쳤고, 리빌 후 700ms~1000ms 대기한 뒤 다시 캡처해서야 발견함 — `rotateY(180deg)`는 "리빌 후에 보여야 하는 면(front)" 쪽에 걸어야 함. 이후 승/패 하이라이트(초록 링/글로우, 패배 카드 흐림)까지 정상 동작 확인.
  - 카드 뒷면 아이콘을 `🂠`(카드 슈트 유니코드 기호, 폰트 지원이 약해 일부 환경에서 tofu 박스로 렌더링됨)에서 `🎴`(컬러 이모지, 호환성 좋음)로 교체.
  - Playwright(`playwright-core` + 로컬 캐시된 Chromium)로 두 브라우저 컨텍스트를 띄워 방 생성→참가→Ready→카드 선택→결과→재경기 전체 플로우와 다크모드를 검증. `npm run build`/`npm run lint` 모두 통과.
  - ⚠️ `frontend/package.json`에 `tailwindcss`, `@tailwindcss/vite`가 새로 추가됨 — Docker 이미지는 `npm install` 시점에 이 의존성을 받으므로, 다음에 `docker compose up`으로 띄울 때는 반드시 `--build`(또는 `docker compose build frontend`)로 이미지를 다시 빌드해야 함. 그냥 `up`만 하면 기존 익명 볼륨의 구 `node_modules`를 그대로 쓰게 되어 Tailwind가 빠진 채로 뜰 수 있음. (실제로 이번 세션에서 `up --build`만으로는 익명 볼륨이 구 `node_modules`를 물고 있어 `@tailwindcss/vite` 모듈을 못 찾는 에러가 났고, `down -v` 후 재기동해서 해결함 — 41번 메모와 동일한 케이스.)
- ✅ Ready/재경기 상대방 상태 실시간 표시: 서버가 `ready` 필드 변경(로비 Ready, 라운드 종료로 인한 리셋, 재경기 동의) 시마다 새 이벤트 `playersUpdated`(`{ players }`)를 방 전체에 브로드캐스트하도록 `backend/src/socket/index.ts`의 `ready`/`selectHand`/`rematch` 핸들러에 추가(각각 `roomService.setReady`/`finishRound`/`markRematchReady` 직후). `RoomPage.tsx`와 `GamePage.tsx`는 이제 로컬 낙관적 상태 대신 서버가 보내주는 `players` 배열의 `ready` 필드를 그대로 신뢰해서, 상대방이 Ready/재경기 버튼을 누르는 즉시 내 화면에도 배지로 표시됨(`RoomPage`는 Ready/대기 중, `GamePage` 결과 화면은 재경기 동의/대기 중). `CLAUDE.md`의 Server → Client 이벤트 표에도 `playersUpdated` 추가함. Playwright로 "A만 Ready/재경기 클릭했을 때 B 화면에 A의 배지가 뜨는지" 직접 검증 완료.
  - 참고: `removePlayer`(disconnect) 쪽은 이번에 손대지 않음 — 상대방이 연결이 끊겨도 남은 플레이어의 화면에 별도 알림/목록 갱신 이벤트가 없는 기존 갭은 그대로 남아있음 (필요해지면 별도로 다룰 것).
- ✅ 상대방 나감 처리: 위에서 남겨둔 갭을 메움. `backend/src/socket/index.ts`의 `disconnect` 핸들러가 `roomService.removePlayer(socket.id)`의 반환값(남은 플레이어가 있는 `Room`)을 보고 새 이벤트 `playerLeft`(`{ players }`)를 방에 브로드캐스트하도록 수정(방이 완전히 비어 삭제된 경우엔 emit 안 함). `removePlayer`는 기존에도 남은 플레이어들의 `ready`/`selectedHand`를 초기화하고 `gameState`를 `WAITING`으로 되돌리고 있었으므로, 프론트만 이 이벤트를 받아 화면을 전환하면 됐음. `App.tsx`가 전역으로 `playerLeft`를 구독해 현재 화면이 `game`이면 `room` 화면(남은 플레이어만 담긴 목록)으로 강제 전환하고, 상단에 "상대방이 방을 나가서 대기실로 돌아왔습니다" 토스트(4초 후 자동 소멸)를 띄움. 이미 `room` 화면에 있을 때(게임 시작 전 로비 단계에서 상대가 나간 경우)는 `RoomPage`가 직접 `playerLeft`를 `playerJoined`/`playersUpdated`와 동일하게 처리해 목록만 갱신(별도 화면 전환 불필요). `CLAUDE.md` 이벤트 표에 `playerLeft` 추가. Playwright로 두 시나리오(게임 중 상대 브라우저 컨텍스트 종료 / 로비 대기 중 상대 종료) 모두 검증 완료 — 게임 중 이탈 시 남은 플레이어가 대기실로 돌아오고 목록에서 상대가 사라짐, 로비 이탈 시에도 목록에서 즉시 사라짐.
  - `나가기`(GamePage 결과 화면 버튼)로 스스로 나가는 경우는 이번 범위 밖 — 그건 서버에 아무 것도 알리지 않고 프론트만 메인 화면으로 돌아가는 기존 동작 그대로임(소켓 연결은 유지된 채 방에 남아있는 상태). 필요하면 별도로 `leaveRoom` 같은 명시적 이벤트를 추가해야 함.
- ✅ 대기실(RoomPage) 나가기 버튼 추가 + 자발적 나가기 이벤트: 위에서 남긴 갭까지 마저 메움. `backend/src/socket/index.ts`에 `leaveCurrentRoom(io, socket)` 헬퍼를 추출해 새 client→server 이벤트 `leaveRoom`(페이로드 없음, `removePlayer`가 socketId만으로 방을 찾음)과 기존 `disconnect` 핸들러가 동일 로직(플레이어 제거 → `socket.leave(roomCode)` → 남은 인원 있으면 `playerLeft` 브로드캐스트)을 공유하도록 정리. `RoomPage.tsx`에 "나가기" 버튼(Ready 버튼 옆, secondary variant) 추가 — 클릭 시 `leaveRoom` emit 후 `onExit()`으로 메인 화면 복귀. `App.tsx`가 `RoomPage`에 `onExit` prop 전달. 겸사겸사 `GamePage.tsx`의 기존 "나가기" 버튼도 동일하게 `leaveRoom`을 emit하도록 고쳐서, 그동안 서버에 알리지 않고 나가던(방에 유령처럼 남아있던) 기존 버그를 같이 잡음. `CLAUDE.md` Client → Server 이벤트 표에 `leaveRoom` 추가. Playwright로 (1) 대기실에서 나가기 클릭 시 본인은 메인 화면으로, 상대방 화면에선 즉시 목록에서 사라짐 (2) 나간 직후 같은 방 코드로 제3의 플레이어가 실제로 입장 가능함(서버 쪽 슬롯이 정말로 비워졌는지, 단순 화면 전환이 아닌지)을 확인 완료.

- ✅ 상대방 카드 소모 현황 공개(전략적 선택을 위해, 사용자 확정): 기본카드(가위/바위/보)는 상대방 것도 정확한 수량 그대로 보여주고, 특수카드(총/중지/거울)는 어떤 종류인지는 숨기되 **총 보유 개수만** 공개 — 상대가 뭘 이미 썼는지 보고 전략은 짤 수 있되, 특수카드의 "무슨 카드인지 모르는" 긴장감은 낼 때까지 유지. 이건 단순 프론트 표시 문제가 아니라 **서버가 애초에 상대방의 정확한 특수카드 구성을 클라이언트에 보내지 않도록** 막아야 진짜 의미가 있음(안 그러면 브라우저 개발자 도구 네트워크 탭에서 그냥 다 보임) — 그래서 `players`가 실리는 모든 이벤트(`playerJoined`/`playersUpdated`/`playerLeft`/`gameStarted`)를 방 전체 브로드캐스트 한 번에서 **수신자별 개별 emit**으로 바꿈. `backend/src/socket/index.ts`에 `toWirePlayers(players, viewerSocketId)` 헬퍼를 추가해 본인 항목은 원본 그대로, 상대방 항목은 `cards.gun/middleFinger/mirror`를 0으로 지우고 대신 `specialCardCount`(총 특수카드 개수, 서버가 계산)를 추가해서 반환하도록 함. 이 변환 결과를 `broadcastPlayers(io, room, event, extra?)` 헬퍼로 각 플레이어의 소켓에 개별 emit(`io.to(socketId).emit(...)`)하도록 기존 `io.to(roomCode).emit(...)` 호출들을 전부 교체. `frontend/src/types/index.ts`의 `Player`에 `specialCardCount: number` 추가, `GamePage.tsx` 헤더에 "{상대닉네임} 카드 ✊N ✌️N ✋N 🎴×N" 상시 표시 줄 추가(라운드마다 갱신되는 걸 계속 볼 수 있게 헤더에 고정 배치). Playwright로 상대가 라운드에서 진 뒤 그 카드 수량은 정확히 -1, 특수카드 총량은 +1 되는 게 실시간으로 보이는지 확인.
- ✅ 기본카드 시작 수량 3장 → **2장**으로 축소(`game/rps/rps.cards.ts`의 `STARTING_BASIC_COUNT`). 위에서 추가한 상대 카드 공개 기능과 맞물려서, 기본카드 폭이 좁아진 만큼 상대가 뭘 이미 썼는지 보고 남은 선택지를 좁혀나가는 전략성이 더 커짐(수학적으로 여전히 매치 도중 카드가 0장까지 갈 수 없다는 이전 불변식도 그대로 유지 — 승리 시 소모 최대치는 `winsToMatch`(2~3)뿐이라 시작 수량 5장(기본 2×2+특수 1)보다 항상 작음).
- ✅ 3연속 무승부로 카드가 초기화될 때 화면에 안내 문구 추가(그동안은 서버에서 조용히 리셋만 되고 클라이언트엔 아무 표시가 없었음). `room.service.ts`의 `applyRoundResult` 반환 타입을 `Room` 단독에서 `{ room, cardsReset: boolean }`로 바꿔서 이번 라운드가 리셋을 유발했는지 호출부에 알려주도록 함. `socket/index.ts`가 이 값을 `GameResult.cardsReset`에 실어 `result` 이벤트로 전송(`game/index.ts`의 `GameResult`, `frontend/src/types/index.ts`의 `GameResult`에 각각 필드 추가). `GamePage.tsx` 결과 화면에 `result.cardsReset`이 참이면 안내 배지를 라운드 결과 문구 밑에 표시. `CLAUDE.md`의 `result` 이벤트 페이로드 표에도 `winsDelta`/`cardsReset` 반영(그동안 누락돼 있던 `winsDelta`도 이 김에 보완). Playwright로 (1) 시작 기본카드가 실제로 2장인지 (2) 결정 라운드로 먼저 카드를 하나 소모시킨 뒤 (3) 같은 카드로 3연속 무승부를 유도해서 리셋 안내가 뜨는 동시에 소모됐던 카드 수량이 정확히 원상복구(2장)되는지까지 확인.
- ✅ 무승부도 카드를 소모하게 변경 + 리셋 트리거를 "3연속 무승부" → "누적 3회 무승부"로 바꾸고, 무승부 스택을 화면에 상시 표시(사용자와 대화로 설계 확정: 무승부가 공짜였던 걸 없애 예측 가능한 패가 상대와 겹치면 손해를 보게 만들어 전략성을 높이되, "연속"이 아니라 "누적"으로 바꿔서 무승부가 띄엄띄엄 나도 반드시 3번째에 안전장치가 걸리도록 함 — 계산상 리셋 사이 최악의 소모량이 항상 시작 보유량 이내라 카드 고갈 걱정 없음).
  - `game/rps/rps.cards.ts`: `CONSECUTIVE_DRAWS_TO_RESET` → `DRAWS_TO_RESET`으로 개명(값은 3 그대로, 의미만 "누적"으로 바뀜).
  - `room.types.ts`의 `Room.consecutiveDraws` → `Room.drawStack`으로 개명.
  - `room.service.ts`의 `applyRoundResult`: 카드 소모를 무승부/결정 라운드 분기 이전으로 옮겨서 **양쪽 다 낸 카드를 항상 소모**하도록 변경(기존엔 무승부면 소모를 건너뜀). 무승부 분기에서 `drawStack`을 증가시키고 임계값 도달 시 카드 리셋 + `drawStack=0`. 결정 라운드 분기에서는 더 이상 `drawStack`을 0으로 되돌리지 않음(누적 유지가 핵심 변경점). `markRematchReady`가 새 매치를 시작하는 분기에서 `drawStack`도 0으로 초기화하도록 추가(예전엔 결정 라운드가 항상 카운터를 0으로 되돌려서 매치가 끝날 때 자동으로 0이었지만, 누적 방식에서는 더 이상 그렇지 않아 명시적으로 리셋해줘야 함).
  - `socket/index.ts`: `gameStarted`에 `drawStack` 추가, `result`(`GameResult`)에 `drawStack` 추가, 그동안 페이로드 없이 보내던 `rematchStarted`에도 `{ drawStack }`을 실어 보내도록 변경(새 매치 시작 시 0으로 리셋된 값을 프론트가 클로저 추측 없이 바로 받도록 — `gameStarted` 때 썼던 것과 같은 패턴).

  - Playwright로 (1) 무승부 라운드에서 실제로 카드가 1장 소모되는지 (2) 결정 라운드가 껴도 스택이 리셋되지 않고 그대로 유지되는지(연속이 아니라 누적임을 증명) (3) 스택이 2까지 쌓였다가 3번째 무승부에서 리셋 안내와 함께 카드가 원상복구되는지까지 확인. 헤더의 점 인디케이터가 라운드마다 정확히 채워지는 것도 스크린샷으로 확인.
- ✅ 같은 공유기 내 다른 기기(LAN)와 실제로 플레이 테스트할 수 있도록 설정 정리. 예전엔 `docker-compose.yml`에 `VITE_BACKEND_URL=http://localhost:3000`, `CLIENT_ORIGIN=http://localhost:5173`이 하드코딩돼 있어서, 호스트가 아닌 다른 기기에서 접속하면 그 기기 자신의 localhost를 찾다가 실패(CORS/연결 실패)했음 — 사용자가 "예전에 테스트가 안 됐다"고 기억하던 원인. `docker-compose.yml`을 `${VITE_BACKEND_URL:-http://localhost:3000}` / `${CLIENT_ORIGIN:-http://localhost:5173}` 형태로 바꿔 기본값(솔로 개발 시 localhost)은 그대로 유지하면서, gitignore된 루트 `.env` 파일(이 PC 전용, git엔 안 올라감)에 실제 LAN IP를 넣어 오버라이드하도록 함. 호스트 PC의 이더넷 어댑터 IP를 `Get-NetIPConfiguration`으로 확인(`175.195.8.97`, 서브넷 175.195.8.0/24, Windows 네트워크 프로필 Private — 방화벽 이슈 가능성 낮음 확인). 호스트도 `localhost` 대신 같은 LAN IP로 접속해야 두 기기의 Origin이 동일해져서 CORS가 안 걸림. Playwright로 `http://175.195.8.97:5173`을 통해 실제로 소켓 연결·방 생성이 되는지 확인 완료, 이후 사용자가 실제 모바일 기기로 접속해서 정상 동작 확인함(모바일 반응형 레이아웃도 문제없이 잘 나옴). 원격지(다른 네트워크)에서 테스트하려면 ngrok 같은 터널링이 별도로 필요하다는 점은 안내만 하고 아직 설정 안 함.
- ✅ README.md를 초기 MVP 상태에서 현재 상태로 전면 갱신 — 실제 플레이 화면 스크린샷 5장(`docs/screenshots/`, Playwright로 촬영: 메인/대기실/카드선택/특수카드 규칙/결과), 현재 게임 규칙 전체(매치 방식 선택, 특수카드 매트릭스, 카드 소모/보상/무승부 스택 리셋), 실제 소스 구조를 반영한 프로젝트 트리 추가.
- ✅ 가위/바위/보 표시 순서를 한국어 관용 순서(**가위→바위→보**)로 통일 — 지금까지 `frontend/src/game/rps/cards.ts`의 `HAND_CARDS`가 영어식(rock→scissors→paper) 순서였고, `MainPage.tsx` 타이틀 이모지(✊✌️✋)와 `GamePage.tsx`의 상대 카드 현황 줄도 같은 순서를 따르고 있었음 — 전부 가위(✌️)→바위(✊)→보(✋) 순으로 재정렬. 특수카드 순서(총→중지→거울, `GamePage.tsx`의 `<details>` 안내 이모지 🔫🖕🪞)는 이미 대응 순서가 맞아서 손 안 댐. `CLAUDE.md`의 `Hand` 타입 설명에 "화면에 손 모양을 나열할 땐 항상 가위→바위→보 순서를 따른다"는 규칙을 명시해서 앞으로 추가되는 UI도 이 순서를 기본으로 따르도록 함(사용자가 앞으로도 이 순서를 계속 쓰겠다고 확정). Playwright로 메인 화면 타이틀, 손패 선택 그리드, 상대 카드 현황 줄 전부 가위-바위-보 순서로 뜨는지 확인.
